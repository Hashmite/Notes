

# Git一文带你理解如何使用

* 工作区: 在电脑上能看到的目录
* 版本库: 工作区有一个隐藏目录 .git ， 这个不算工作区，而是git的版本库
* 暂存区: 一般存放在 .git 目录下的index 文件 (.git/index) 中，暂存区有时也叫索引

![](https://i-blog.csdnimg.cn/direct/ef78d80ebe9d4d5ca31eadec6a921cd9.png)

## 基本操作(重点基础操作)
![](https://i-blog.csdnimg.cn/direct/74be59aa5d4946ec87ccc4f5de6c031d.png)
* git init    初始化 git 仓库
* git add  .  将所有文件添加到暂存区中
* git commit -m "frist commit"  添加注释
* git remote add orgin git@github.com:**/test.git   将本地库(local repository) 与远程仓库 （remote repository） 进行连接
* git pull origin master  上传之前，需要pull 跟新一下，将远程仓库中的文件更新到本地仓库中
* git push -u origin master  推送到远程仓库中

【流程】 （workspace）工作区 --> 添加到 （stagin）暂存区 --> 提交到 （local repository） 本地库（版本库）--> 推送至 （remote repositroy） 远程仓库中
* git fetch   从远程花去最新到本地，不会进行合并操作
* git pull (相当于idea 中 的 update)  从远程抓取最新到本地，并进行合并（merge）（git pull = git fetch + git merge）

merge:  作用是分支合并，将其他分支的内容合并到当前分支上
*开发分支 （dev）  上的代码达到上线的标准后，要合并到master 分支上
1. git checkout dev
2. git pull (origin master)
3. git checkout master  已经切换到了主分支上
4. git merge dev
5. git push (-u origin master)


1. 查看分支  git branch -a/-r

## 分支切换与隐藏
切换分支遇到问题
        当正常使用git切换分支时，出现一线提示（请在切换分支之前你的更改或者隐藏他们）
解决方案：
1. 因为当前分支dev 最初也是从master 分支上衍生出来的，而此时你要从该分支上切换到其他主分支，那么你需要先把该dev 分支的改动提交后才能切换， 但是该dev 分支上还没有完成的全部的修改，你不想提交，那么此时你就要选择 stash（存放）他们（你当前分支上的改动的却没有提交commit 的内容）.所以第一步，在当前分支上执行git stash 命令，将当前分支存起来， id 为 807be186826
2. 这个时候再执行 git status ，。显示没有东西需要提交。接着就可以主分支master 上创建并切换到新的分支上修复另外一个bug, 修改玩这个bug 后也要提交后，就该回到dev,分支上区继续未完成的bug. 执行 git checkout dev 切换到dev 分支，这个时候执行git status 依旧显示没有东西需要提交。比较我们前边已经成功将dev 上未提交的改动给  “隐藏”了，这时， 用git stash list 查看 “存储”  的列表， 会发现id 807be186826 在 存藏 项目爱列表 中，我们需要将其恢复
           2.1 git stash pop 。 恢复的同时把stash  存储列表的内容也删了。这时候在执行git stash list 。id 为 807be186826  的储藏 项目不会再列表中

**第二步**：把虚拟DOM 渲染为真实DOM（浏览器页面中最后 渲染出阿里，让用户看见的ODM 元素）

> 补充说明： 第一次渲染页面是直接从虚拟DOM -> 真实DOM，但是后期视图更新的时候，需要经过一个DOM-DIFF 的对比，计算出补丁包PATCH （也就是两次视图差异部分）进行渲染

1. 第一次渲染完成后， 保存 虚拟DOM 缓存=> oldVirtualDOM
2. 按照最新的数据，把jsx重新编译为 ‘全新的virtualDom’ (全部重新编译一遍)，拿新的虚拟DOM 和之前缓存起来的虚拟DOM 进行对比 -> DOM-DIFF , 之后生成-> PATCH 补丁包， 再把该补丁包渲染



> 总结： jsx 语法通过调用语法包编译并创建createElement，然后执行该函数生成虚拟DOM ，最后通过render 将 虚拟DOM 渲染为真实DOM

## props 基本认识



- **props :** **属性是只读属性，不可以修改，原理是props 对象被冻结了**
  - 关于对象的规则设置
    - 冻结对象： Object.freeze(obj); 检查是否被冻结： Object.isFrozen(obj) => true/false

      - 检查被密封: Object.seal(obj) ; 检查是否被密封： Object.isSealed(obj)可以修改，其他操作不可做
    - 不可拓展； 把对象设置为不可拓展： Object.preventExtensions(obj)
      - 检查是否可拓展： Object.isExtensible(obj)
        - 被设置不可拓展对象： 除不能新增成员，其余的擦欧总都可以处理
- 作用： 父组件{index.jsx} 调用子组件{DemonOne.jsx}的时候，可以基于属性，把不同的信息传递给子组件，子组件接收响应的属性值，呈现不同的效果，让组件的复用性增强
  - 虽然对于传递过来的属性，不能直接修改，但是可以做一些规则校验
    - 通过把函数当作对象，设置静态的私有属性方法，来给其设置属性的校验规则
      - 通过函数组件.defaultProps = {} 设置
      - 设置其他规则，例如： 数据值格式、是否必传...... 依赖于这个插件 ： prop-types
        - 对应官网： [props](https://github.com/facebook/prop-types)
  - 被冻结对象，即是不可拓展，也是密封的，同理，被密封的对象，也是不可拓展的

## 状态管理

在 React 函数组件中，闭包是 JavaScript 的核心特性，而非 React 的缺陷。它导致“事件处理函数捕获的是定义时的 state 快照”，这是理解 useState 行为的关键。以下用最简示例精准解析：



> 🔑 闭包本质一句话
>
> **每次组件渲染 = 生成全新作用域**
> 该次渲染中定义的函数（如事件处理）会“记住”此作用域内的所有变量（含 state），形成**独立闭包**。后续 state 更新不会影响已捕获的旧值，Render（协调） + Commit（提交） = 完整渲染周期

比如这里的定时任务三秒后该handle 方法已经执行完毕（一个闭包），并提交更新视图:

```tsx
    const [x, setX] = useState(0)
    console.log('视图更新渲染' )
    const handle = () => {
        console.log('[点击时] 当前 count =', x); // 捕获点击瞬间的 count
        setTimeout(() => {
            setX(x + 1); // ❌ 闭包陷阱：count 始终是点击时的旧值
            console.log('[3秒后] 使用的 count =', x);
        }, 3000);

    }
```

  - 但是具备属性及插槽,父组件可以控制其重新渲染,渲染流程简单,渲染速度较快!  基于FP(函数式编程)思想设计,提供更细粒度的逻辑组织和复用!
- **类组件**
  - **具备:  “状态、ref、周期函数、属性、插槽”等内容**,可以灵活的控制组件更新,基于钩子函数也可灵活掌控不同阶段处理不同的事情!
  - 渲染流程繁琐,渲染速度相对较慢!基于OOP(面向对象编程)思想设计,更方便实现继承等!



### setState 进阶（类组件）✅

> 在 React 中，**`setState` 仅适用于类组件（Class Components）**，用于安全、高效地更新组件内部状态
>
> | 场景                              | 说明                                                         |
> | :-------------------------------- | :----------------------------------------------------------- |
> | **类组件中需要更新 `this.state`** | 唯一合法使用场景（如表单输入、异步数据加载后更新）           |
> | **需在状态更新后执行逻辑**        | 使用 `setState(updater, callback)` 的回调参数                |
> | **依赖前一状态计算新值**          | 使用函数式更新：`setState(prev => ({ count: prev.count + 1 }))` |

**this.setState({partialState},{callback})**

- **partialState**: 支持局部状态更改 

- **callback**: 在状态更改/视图更新完毕后执行（也可以说只要执行了setState， callback 一定会执行）

  - callback 发生在componentDidUpdate 周期函数之后（componentDidUpdate 会在任何状态更改后都触发执行，而回调函数方式，可以在指定状态更新后处理）
  - 特殊，即使我们基于shouldComponentUpdate阻止了状态/视图更新，componentDidUpdate 周期函数肯定不会执行，但是我们设置的这个Callback 回调函数依然会被触发执行

  ```jsx 
  this.setState({
      x:100  // 不论总共有多少状态，我们只修改了x,其余的状态不动
  })
  ```

  ```jsx
  class Demo extends React.Component {
    state={
        x:10,
        y:4,
        z:1
    };
    handle=()=>{
        let{x,y,z} = this.state;
        this.setState({x:x+1});
        console.log(x);
        this.setState({y:y+1});
        console.log(y);
        this.setState({z:z+1});
        console.log(z);
    }
    render() {
        console.log("视图渲染")
        let {x,y,z} = this.state;
        return<div>
            x:{x}   -   y:{y}   -   z:{z}
            <br/>
            <button onClick={this.handle}>按钮</button>
        </div>
    }
  }
  ```



如上 9 ~15行代码；当触发事件调用到该处函数时， 不会立即更新状态和视图而是将他加入到更新队列中，**当上下文中的代码都处理完毕后**，才会让更新队列中的任务，统一渲染或者更新一次（批处理）：

 重要注意事项（类组件专属）

1. **异步与批量更新**
   `setState` 可能异步执行（合成事件/生命周期中），**切勿依赖 `this.state` 立即生效（上下文处理完毕后，才更新队列）**
   → 始终用函数式更新：`setState(prev => ...)`
2. **禁止在 `render` 中调用**
   会导致无限循环渲染（React 18+ 会抛出错误）
3. **浅合并特性**
   `setState({ a: 1 })` 不会覆盖 `state.b`，但嵌套对象需手动展开
4. **回调参数替代 `componentDidUpdate`** （就算componentDidUpdate` 被暂停了，callback 也可以回调）
   适合更新后立即操作 DOM（如滚动定位）

| 阶段 | 生命周期                          | 说明                                     | ⚠️ 关键注意事项                                               |
| ---- | --------------------------------- | ---------------------------------------- | ------------------------------------------------------------ |
| 1    | `static getDerivedStateFromProps` | 每次渲染前调用（含 setState 触发的更新） | ❌ 禁止调用 `setState`（静态方法，设计上通过返回对象更新 state） |
| 2    | `shouldComponentUpdate`           | 决定是否继续更新（性能优化关键）         | ❌ 禁止调用 `setState`（会触发无限循环） ✅ 返回 `false` 可跳过后续所有流程 |
| 3    | `render`                          | 生成新的 Virtual DOM                     | ❌ 绝对禁止调用 `setState`（React 会抛出 `Cannot update during an existing state transition` 错误） |
| 4    | `getSnapshotBeforeUpdate`         | DOM 更新前捕获快照（如滚动位置）         | ❌ 禁止调用 `setState`（React 会警告：`Cannot update state in getSnapshotBeforeUpdate`） |
| 5    | `componentDidUpdate`              | DOM 已更新，可执行副作用                 | ✅ 可调用 `setState`，但必须加条件判断 ❌ 无条件调用 → 无限循环 |
| 6    | `setState(updater, callback)`     | 回调在 `componentDidUpdate` 之后执行     | ✅ 安全获取更新后 DOM 状态                                    |

当我们需要根据前两个参数更新后的值作为第三个参数的值时，这个时候react 提供了一个方法， 来自于 react-dom 中解构出来的 **flushSync**

- flushSync: 可以刷新 "updater 更新队列",**强制中断当前批量更新上下文，同步执行并提交更新到 DOM**！！！

核心机制澄清（破除误解

| 误解说法            | 正确解释                                                     |
| :------------------ | :----------------------------------------------------------- |
| “立刻批处理一次”    | ❌ **打断批处理**：强制清空当前更新队列中 `flushSync` 回调内的更新，并**立即同步渲染** |
| “刷新 updater 队列” | ⚠️ **仅清空回调内更新**：仅处理 `flushSync` 回调中触发的更新，外部更新仍按原批量逻辑执行 |
| “安全高效”          | ⚠️ **阻塞主线程**：同步执行 → 阻塞浏览器渲染/用户交互，滥用会导致卡顿 |

关键行为演示（React 18+）

```jsx
import { flushSync } from 'react-dom';

function Demo() {
  const [count, setCount] = useState(0);
  const divRef = useRef(null);

  const handleClick = () => {
    // 普通更新（加入批量队列）
    setCount(1); 
    
    // ❗ flushSync 强制中断批量：
    flushSync(() => {
      setCount(2); // 立即同步渲染 → DOM 更新为 2
    });
    
    // 此时可安全读取更新后的 DOM
    console.log(divRef.current.textContent); // 输出 "2"
    
    // 外部更新（count=1）被推迟到 flushSync 之后执行
    // 最终 DOM 显示 "1"（因后续更新覆盖）
  };

  return (
    <div ref={divRef} onClick={handleClick}>
      {count}
    </div>
  );
}
```

```sequence


    participant  用户点击
    participant  React 批量上下文
    participant  flushSync
    participant  DOM
    
    用户点击->React 批量上下文: 触React 批量上下文 handleClick
    React 批量上下文->>React 批量上下文: setCount(1) → 入队（待批量）
    React 批量上下文->>flushSync: flushSync(() => setCount(2))
    flushSync->>DOM: 立即同步渲染 count=2
    flushSync-->>React 批量上下文: 返回（DOM 已更新）
    React 批量上下文->>React 批量上下文: 读取 DOM（此时为 2）
    React 批量上下文->>DOM: 批量处理剩余更新（count=1）
    Note over DOM: 最终显示 1（被覆盖）
```

使用 `UNSAFE_componentWillReceiveProps` 等方法会触发控制台警告，且 **React 19 可能彻底移除**。迁移指南：

- 用 `getDerivedStateFromProps` 替代 `componentWillReceiveProps`
- 用 `getSnapshotBeforeUpdate` + `componentDidUpdate` 替代 `componentWillUpdate`

#### 各生命周期中调用 `setState` 安全指南

| 生命周期               | 是否可调用`setState`  | 原因/风险                                                    |
| :--------------------- | :-------------------- | :----------------------------------------------------------- |
| `constructor`          | ❌                     | 组件未挂载，`this.state` 未初始化                            |
| `componentDidMount`    | ✅（需条件）           | 安全，但会触发额外渲染（通常可接受）                         |
| `componentDidUpdate`   | ✅（**必须条件判断**） | 无条件调用 → 无限循环（例：`if (prevProps.id !== this.props.id)`） |
| `componentWillUnmount` | ❌                     | 组件即将销毁，更新无意义且可能报错                           |
| 事件处理函数           | ✅                     | React 18+ 自动批量更新，安全高效                             |
| `setTimeout`/Promise   | ✅（React 18+）        | React 17- 需 `unstable_batchedUpdates` 手动批量              |

#### 经典陷阱与解决方案

```jsx
// ❌ 陷阱1：componentDidUpdate 无限循环
componentDidUpdate() {
  this.setState({ count: this.state.count + 1 }); // 每次更新触发新更新！
}

// ✅ 修复：添加条件判断
componentDidUpdate(prevProps, prevState) {
  if (prevState.count !== this.state.count) {
    this.setState({ log: `Updated to ${this.state.count}` }); // 安全
  }
}

// ❌ 陷阱2：render 中调用 setState
render() {
  this.setState({ flag: true }); // React 抛出错误！
  return <div>...</div>;
}

// ✅ 正确：在事件中更新
handleClick = () => this.setState({ flag: true });
```

#### 终极总结

1. **`setState` 本身不直接“设计”生命周期**，而是**触发更新流程**，由 React 调度器按规范执行生命周期链

2. **安全调用 `setState` 的位置**：

   - ✅ 事件处理函数、定时器（React 18+）
   - ✅ `componentDidMount`（谨慎）
   - ✅ `componentDidUpdate`（**必须条件判断**）

3. **现代开发强烈建议**：

   > 🌐 **新项目请使用函数组件 + Hooks**
   >
   > - 用 `useEffect` 替代 `componentDidMount`/`Update`/`Unmount`
   > - 用 `useState` 替代 `setState`
   > - 逻辑更清晰，无生命周期陷阱
   >   （React 官方文档已将 Hooks 作为默认方案）

理解 `setState` 与生命周期的交互，是维护遗留类组件的关键；但面向未来，**掌握 Hooks 是现代 React 开发的必备能力**。



### useState 进阶（函数组件）✅	



> React Hook  函数之一；**目的是在函数组件中使用状态**，**并且后期基于状态的修改，可以让组件更新，类似于类组件中的setState**； 他执行于 组件加载完之后， 该干些什么
>
> let [state,setState] = useState(initialValue);   返回一个state,以及更新state 的函数
>

- 执行useState，传递的initialValue （初始状态，可以是任意类型；如果是函数，则会在初始渲染时执行其返回值）是初始状态值，返回结果是一个数组：[状态值，修改状态的方法]

  - state变量存储的是获取的当前状态值
  - setState()变量存储的是修改状态的方法，并且会通知视图更新


**注意：`setState` 本身是异步的，在同一个同步事件中多次调用 `setState` 会被合并（批处理），只触发一次渲染。**

- `useState` 的 **setter（如 `setX`）调用是同步的**，但 **组件重新渲染是异步调度的**。这是 React 精心设计的机制，需精准理解“同步入队”与“异步渲染”的区别

-   也就是说，在同步中多次调用setState，第一次他会把需要更新 值放在更新队列中，直到函数的上下文执行完毕才统一更新更新队列，并通知视图渲染    
  - ​    可以使用flusySync（）强制 更新当前更新队列

#### useState处理机制



> 函数组件的每一次渲染（或者更新），都是把**函数重新执行**，产生全新的“私有上下文”
>
> - 内部的代码都需要重新执行一次，涉及到的函数需要重新构建，这些函数的作用域（私有上下文）是**每一次执行的DEMO 产生的闭包** 
>   -   setState(5)，一般都是直接替换，不合并对象
>   -   setCount(c => c+1)  解决了闭包问题     **优先函数式更新**：`setCount(c => c + 1)` 避免闭包陷阱
> - 每一次执行DEMO 函数，也会把useState 重新执行。
>   - 执行useState，只有第一次，设置的初始值会生效，其余以后再执行，获取的状态都是最新的状态值
>   - 返回的修改状态的方法，每一次都是返回一个新的     
>

#### 内部工作机制（基于 Fiber 架构）

React 16.8+ 引入 Hooks 后，函数组件的状态由 **Fiber 节点** 上的 `memoizedState` 属性维护。每个 Hook 在 Fiber 上以**单向链表**的形式存储：

- 首次渲染时，React 会按调用顺序依次创建 Hook 对象，并挂载到 Fiber 的 `memoizedState` 上。（也叫更新队列），与组件实例解耦
- `dispatch` 本质是向更新队列推入 action，触发 **Scheduler 调度**
- 批量更新由 **React 事件系统 + Scheduler** 协同控制
- 每个 Hook 对象包含：
  - `memoizedState`：当前状态值。
  - `queue`：指向更新队列（环形链表），存放待处理的 `setState` 动作。
  - `next`：指向下一个 Hook。

**关键点**：Hooks 的调用顺序必须稳定，否则会导致状态错乱（这也是为什么不能在条件、循环中调用 Hooks）。

基本源码原理如下代码： 

```js
var _state;
function useState(initialValue){
    if(typeof  _state === "undefined") _state = initialValue;
    var setState=function setState(value){
        _state=value;
    };
    return [_state,setState];
}
```



#### 同步异步处理

`useState` 的 **setter（如 `setX`）调用是同步的**，但 **组件重新渲染是异步调度的**。这是 React 精心设计的机制，需精准理解“**同步入队**”与“**异步渲染**”的区别：

| 操作                      | 同步/异步  | 说明                                               | 验证方式                                  |
| :------------------------ | :--------- | :------------------------------------------------- | :---------------------------------------- |
| **调用 `setX(newValue)`** | ✅ **同步** | 立即将更新推入 React 内部队列，标记 Fiber 需更新   | `console.log('setter executed')` 立即输出 |
| **当前作用域读取 `x`**    | ❌ **不变** | 函数组件是闭包快照，`x` 仍是本次渲染的旧值         | `setX(2); console.log(x)` 仍输出 1        |
| **组件重新渲染**          | ⏳ **异步** | 由 Scheduler 在微任务/宏任务中调度（受批处理影响） | `useEffect` 中才能看到新值                |
| **DOM 更新完成**          | ⏳ **异步** | 渲染后协调 Virtual DOM → 提交真实 DOM              | `useEffect` 执行时 DOM 已更新             |

 执行useState: 把需要的状态信息都放在对象中统一管理

- 执行setState方法的时候，传递的是啥，就把状态“整体”改为啥值，并不会象类组件中的this.setState 一样，支持部分状态的更新
  -   处理办法就是，将存储的值进行解构出来，通常 这样写  …state   
  -   官方的建议是： 需要多个状态，就把useState执行多次即可  

```tsx
 console.log('视图更新渲染')
    const [x, setX] = useState(10)
    const [y, setY] = useState(20)
    const [z, setZ] = useState(30)

    const handle=()=>{
        setX(x + 1);
        setY(y + 1);
       
        setZ(z + 1);
    }
```

1. 上述执行handle  方法，实际上因为useState 是异步函数，所以 上诉代码中的setXxx被执行后，被依次将值放在了更新队列中，当当前方法或者函数执行完毕后 更新 更新队列并通知视图渲染数据

   -  如果需要单独更新某个值实现同步更新，可以使用 

   - ```tsx
     flushSync(()=>{
         // 需要同步的状态值设置  
     })
     ```

   - 该方法的作用就是执行flushSync ，并立刻刷新更新队列并通知视图渲染

2. 在react16 中, 也和this.setState一样，放在合成事件&周期函数中，都是异步的操作，但是放在其他的异步操作中，（列如： 定时器，手动绑定的事件中等，它是同步的）

🌐 批处理机制（决定“渲染次数”的关键）

| React 版本    | 事件回调内 | setTimeout/Promise | 原生事件   | 说明                       |
| :------------ | :--------- | :----------------- | :--------- | :------------------------- |

  setCount(2);                    // ✅ 同步入队（与上一条合并）
  console.log('3. 再次读:', count);    // 0

  console.log('4. useEffect 中:', count); // 2（批处理后最终值）

###   useReducer 实现对状态统一管理✅

> useReducer 是useState 的升级，而sueState 是 state 的升级
>
> 使用 `useReducer` **结合 Context API** 可实现轻量级、类型安全的跨组件状态统一管理（“简易 Redux 模式”）。简单的说，就是useReducer 将 状态管理和 执行的函数剥离出来由客户端自己维护

```tsx
const [state, dispatch] = useReducer(reducer, initialArg, init?);
```

- **`reducer`**：纯函数 `(state, action) => newState`，根据当前状态和 action 返回新状态。
- **`initialArg`**：初始状态的值，或者传递给 `init` 函数的参数。
- **`init`**（可选）：惰性初始化函数 `(initialArg) => initialState`，返回真正的初始状态。如果提供，则 `initialArg` 会传入 `init`。
- **返回值**：`[state, dispatch]`，`state` 是当前状态，`dispatch` 用于触发状态更新。

> 💡 核心认知
> useReducer 的本质是：将状态更新逻辑从组件中剥离，交由纯函数（reducer）集中管理，并通过 React 的调度系统实现高效、可预测的状态流转。
> 它不是“异步更新”，而是 “同步入队 + 异步调度 + 批量处理” 的精密协作。理解其与 Fiber 架构、Concurrent Rendering 的关联，是掌握现代 React 状态管理的关键。

#### 内部工作机制（基于 Fiber 架构）

`useReducer` 与 `useState` 共享同一套底层机制：它们都存储在 **Fiber 节点** 的 `memoizedState` 属性所指向的 **Hooks 链表** 中。

每个 Hook 在链表中的结构大致如下（简化）：

```tsx
type Hook = {
  memoizedState: any;      // 当前状态值
  baseState: any;          // 基础状态（用于计算更新）
  baseQueue: Update<any> | null; // 待处理的更新队列
  queue: UpdateQueue<any> | null; // 更新队列（包含 pending 环形链表）
  next: Hook | null;       // 指向下一个 Hook
};
```

对于 `useReducer`，`queue` 存储的是该 Hook 专属的更新队列，其中包含：

- `pending`：指向一个环形链表，每个节点代表一次 `dispatch` 触发的更新（包含 action 和优先级等信息）。
- `dispatch`：绑定到当前 Fiber 和 Hook 的分发函数。

#### 更新流程

当调用 dispatch(action) 时，会执行以下步骤：
1. 创建更新对象：将 action 包装成一个更新节点（Update），并附加当前渲染的优先级信息。
2. 入队：将该更新节点加入对应 Hook 的更新队列（queue.pending 环形链表）。
3. 调度更新：标记该 Fiber 节点需要更新，请求调度（Scheduler）。
4. 重新渲染：React 在合适时机（如浏览器空闲）进入渲染阶段，从根节点开始协调。当执行到该函数组件时，会：
◦ 遍历更新队列，依次将每个 action 传入 reducer，计算出新的状态。
◦ 将计算后的新状态存入 memoizedState。
◦ 如果使用了 init 进行惰性初始化，首次渲染时会调用 init(initialArg) 得到初始状态。
5. 渲染组件：用新状态调用函数组件，生成新的 React 元素树。
6. 提交阶段：将变化应用到真实 DOM。
关键点：
• 多次 dispatch 在同一个事件循环中会被合并（批处理），只触发一次渲染。
• dispatch 函数本身是稳定的，不会在渲染间改变（除非组件被销毁重建）。

#### 与 useState 的关系

`useState` 实际上是 `useReducer` 的一个特例，其内部实现可以看作：

````tsx
function useState(initialState) {
  return useReducer(
    (state, action) => typeof action === 'function' ? action(state) : action,
    initialState
  );
}
````

- `useState` 的 `setState` 传入函数时，相当于 `dispatch` 了一个函数 action；传入非函数时，相当于直接替换状态。
- `useReducer` 则更通用，允许自定义 action 类型和更新逻辑。

#### 惰性初始化

useReducer 接受可选的第三个参数 init，用于延迟创建初始状态。这有助于将初始化逻辑提取出来，且只在首次渲染时执行：

```tsx
function init(initialCount) {
  return { count: initialCount };
}
const [state, dispatch] = useReducer(reducer, initialCount, init);
```

如果未提供 `init`，则 `initialArg` 直接作为初始状态；若 `initialArg` 是函数，在 `useState` 中会被调用，但在 `useReducer` 中**不会**（除非通过 `init`）

与 `useState` 相同，`useReducer` 的 `dispatch` 在 React 18+ 中默认自动批处理（包括异步回调）。如果需要立即更新，可以使用 `flushSync`：

#### 性能优化与注意事项

- **reducer 必须是纯函数**：不应有副作用，每次输入相同应返回相同输出。
- **状态不可变性**：reducer 中必须返回新对象，而不是修改原状态。
- **避免在 reducer 中执行异步操作**：所有副作用应放在事件处理或 `useEffect` 中。
- **dispatch 的稳定性**：`dispatch` 在组件的生命周期内保持不变，因此可以安全地传递给子组件，不会导致不必要的重渲染。
- **与 Context 结合**：常与 `useContext` 配合实现跨组件状态共享，将 `dispatch` 通过 context 传递下去，子组件可以直接触发更新。

#### 总结

`useReducer` 是 React 提供的强大状态管理工具，其内部利用 Fiber 上的 Hook 链表和更新队列来管理状态变更。它通过 `dispatch` 触发 action，经 reducer 计算新状态，最终驱动组件重新渲染。理解其机制有助于编写更可预测、更高效的 React 组件，尤其是在处理复杂状态时。

如果需要深入研究，可以查阅 React 源码中 `ReactFiberHooks.js` 中 `useReducer` 的实现，重点关注 `mountReducer` 和 `updateReducer` 两个阶段。

### useReducer+useContext跨组件共享状态✅



> Reducers 可以让你整合组件的状态更新逻辑。Context 允许你将信息传递给更底层的其他组件。你可以将 Reducers 和 Context 结合使用，来管理复杂屏幕的状态,每种方式都有其适用场景和底层机制,这种模式常被称为“轻量级 Redux”，适用于中小型应用
>
> - **核心哲学**: 状态应尽可能靠近使用它的组件。仅当必要时，才提升或全局化。
>
>   ​                                                                                                        — React 官方设计原则

核心思想

- **`useReducer`**：管理复杂的状态逻辑，通过 `dispatch` 触发更新，保证状态变化的可预测性。
- **`useContext`**：创建全局的上下文对象，让深层组件无需 props 就能直接访问共享数据。

两者结合后，我们可以在顶层组件中通过 `useReducer` 创建状态和 `dispatch` 函数，然后通过 `Context.Provider` 向下传递，任何子组件都能消费这些值并触发更新。

最佳思路实践：

- 一个父组件 `App`，负责管理状态。
- 两个子组件：`CounterDisplay` 显示当前计数，`CounterButtons` 提供增减按钮。
- 状态和 `dispatch` 分别通过不同的 Context 传递，以避免不必要的重渲染。

1. 创建 Context 文件（通常单独定义）

   ```tsx
   // store.js
   import React from 'react';
   
   // 分别创建 State 和 Dispatch 的 Context
   export const CountStateContext = React.createContext();
   export const CountDispatchContext = React.createContext();
   ```

2. #### 定义 Reducer 和初始状态

   ```tsx
   // reducer.js
   export const initialState = { count: 0 };
   
   export function countReducer(state, action) {
     switch (action.type) {
       case 'increment':
         return { count: state.count + 1 };
       case 'decrement':
         return { count: state.count - 1 };
       default:
         throw new Error(`Unhandled action type: ${action.type}`);
     }
   }
   ```

3. #### 在顶层组件中使用 Provider 提供状态和 dispatch

   ```tsx
   // App.js
   import React, { useReducer } from 'react';
   import { CountStateContext, CountDispatchContext } from './store';
   import { countReducer, initialState } from './reducer';
   import CounterDisplay from './CounterDisplay';
   import CounterButtons from './CounterButtons';
   
   function App() {
     const [state, dispatch] = useReducer(countReducer, initialState);
   
     return (
       <CountStateContext.Provider value={state}>
         <CountDispatchContext.Provider value={dispatch}>
           <div>
             <h1>跨组件状态共享示例</h1>
             				 	<CounterDisplay />
             <CounterButtons />
           </div>
         </CountDispatchContext.Provider>
       </CountStateContext.Provider>
     );
   }
   
   export default App;
   ```

   

4. #### 在子组件中通过 useContext 消费状态和 dispatch

   - ####     (**显示组件**：只读取状态，不触发更新。)

   ```tsx
   // CounterDisplay.js
   import React, { useContext } from 'react';
   
   import { CountStateContext } from './store';
   
   function CounterDisplay() {
     const state = useContext(CountStateContext);
     return <p>当前计数：{state.count}</p>;
   }
   
   export default CounterDisplay;
   ```

   -  （    **操作组件**：只获取 `dispatch` 函数，用于触发更新 ）

   ```tsx
   // CounterButtons.js
   import React, { useContext } from 'react';
   import { CountDispatchContext } from './store';
   
   function CounterButtons() {
     const dispatch = useContext(CountDispatchContext);
   
     return (
       <div>
         <button onClick={() => dispatch({ type: 'increment' })}>+1</button>
         <button onClick={() => dispatch({ type: 'decrement' })}>-1</button>
       </div>
     );
   }
   
   export default CounterButtons;
   ```

   ### 工作原理剖析

   1. **Context 的创建**：`React.createContext()` 生成两个独立 Context 对象，它们各自包含 `Provider` 和 `Consumer`。在底层，每个 Context 对象对应 Fiber 节点上的一个 context 依赖记录。
   2. **Provider 注入**：`App` 组件渲染时，通过 `CountStateContext.Provider` 和 `CountDispatchContext.Provider` 将 `state` 和 `dispatch` 注入到组件树中。这些值会存储在 Fiber 节点关联的 `context` 结构中。
   3. **消费机制**：
      - 当 `CounterDisplay` 调用 `useContext(CountStateContext)`，React 会向上查找最近的 `CountStateContext.Provider`，获取其 `value`（即当前 `state`）。
      - 同样，`CounterButtons` 通过 `useContext(CountDispatchContext)` 获取 `dispatch`。
   4. **更新流程**：
      - 点击按钮 → 调用 `dispatch({ type: 'increment' })`。
      - `useReducer` 内部将 action 加入更新队列，并调度重新渲染。
      - `App` 组件重新执行，`useReducer` 返回新的 `state`。
      - 由于 `state` 变化，`CountStateContext.Provider` 的 `value` 发生变化，React 会标记所有**消费了 `CountStateContext` 的组件**（即 `CounterDisplay`）为“需要更新”。
      - 这些组件随后重新渲染，获取最新的 `state.count`。
      - 注意：`CountDispatchContext.Provider` 的 `value`（即 `dispatch` 函数）在组件生命周期内**稳定不变**，因此消费它的组件（`CounterButtons`）不会因为状态变化而重新渲染。这就是我们将 `state` 和 `dispatch` 分开传递带来的性能优化。

#### 性能优化建议

- **拆分 Context**：如上述代码所示，将状态和 `dispatch` 放在不同的 Context 中，避免所有消费 `dispatch` 的组件因状态变化而重渲染（`dispatch` 本身不变）。

- **使用 `useMemo` 优化 value**：如果 Provider 的 value 是对象或数组，可以考虑用 `useMemo` 包裹，避免因父组件重渲染导致 value 引用变化，引发不必要的消费者重渲染。

  ```tsx
  const contextValue = useMemo(() => ({ state, dispatch }), [state, dispatch]);
  ```

  但由于我们已经拆分 Context，这一步不是必须的。

- **组件自身优化**：对于消费状态的组件，如果只依赖部分状态，可以使用 `React.memo` 配合自定义比较逻辑，或者将状态进一步拆分。

##### 2. `createSlice` - 定义状态切片

> 接收一个包含名称、初始状态、reducer 函数的对象，自动生成对应的 action creators 和 action types。

```tsx
import { createSlice } from '@reduxjs/toolkit'

const counterSlice = createSlice({
  name: 'counter',
  initialState: { value: 0 },
  reducers: {
    increment: (state) => {
      state.value += 1   // 直接“修改” state（Immer 会处理不可变更新）
    },
    decrement: (state) => {
      state.value -= 1
    },
    incrementByAmount: (state, action) => {
      state.value += action.payload
    },
  },
})

// 导出生成的 action creators
export const { increment, decrement, incrementByAmount } = counterSlice.actions
// 导出 reducer，用于配置 store
export default counterSlice.reducer
```

##### 3. `createAsyncThunk` - 处理异步逻辑

> 创建一个异步 thunk，它会自动派发 pending / fulfilled / rejected 三种状态的 action。

```tsx
import { createAsyncThunk, createSlice } from '@reduxjs/toolkit'
import { fetchUserById } from './api'

// 异步 thunk
export const fetchUser = createAsyncThunk('users/fetchById', async (userId) => {
  const response = await fetchUserById(userId)
  return response.data
})

const usersSlice = createSlice({
  name: 'users',
  initialState: { entities: [], loading: 'idle' },
  reducers: {},
  extraReducers: (builder) => {
    builder
      .addCase(fetchUser.pending, (state) => {
        state.loading = 'loading'
      })
      .addCase(fetchUser.fulfilled, (state, action) => {
        state.loading = 'idle'
        state.entities.push(action.payload)
      })
      .addCase(fetchUser.rejected, (state) => {
        state.loading = 'idle'
        // 可处理错误
      })
  },
})
```

## React 生命周期

> **React 组件生命周期是指组件从创建、渲染、更新到卸载的整个过程**。在 React 的不同版本中，生命周期 API 有所变化，但核心概念一致。下面分**类组件**和**函数组件**两部分介绍。

### 类组件生命周期

#### React 16.3 之前（旧版）

- **挂载阶段**
  - `constructor(props)`：初始化 state、绑定方法。
  - `componentWillMount()`（已废弃）：在渲染前调用，服务端渲染会调用。
  - `render()`：必须，返回 JSX。
  - `componentDidMount()`：组件挂载后调用（DOM 已存在），适合做数据请求、订阅、操作 DOM。

- **更新阶段**（由 props 或 state 变化触发）
  - `componentWillReceiveProps(nextProps)`（已废弃）：当 props 改变时调用，用于根据新 props 更新 state。
  - `shouldComponentUpdate(nextProps, nextState)`：返回布尔值决定是否重新渲染，可用于性能优化。
  - `componentWillUpdate(nextProps, nextState)`（已废弃）：渲染前调用，不能在此调用 `setState`。
  - `render()`：重新渲染。
  - `componentDidUpdate(prevProps, prevState)`：更新后调用，可操作 DOM 或发起网络请求。

- **卸载阶段**
  - `componentWillUnmount()`：组件卸载前调用，用于清除定时器、取消订阅等清理工作。
- **错误处理**
  - `componentDidCatch(error, info)`：捕获子组件错误，实现错误边界。

####  React16.3+废弃部分生命周期,引入新方法

React 16.3 废弃了 `componentWillMount`、`componentWillReceiveProps`、`componentWillUpdate`，并引入了两个静态方法：

- **`static getDerivedStateFromProps(props, state)`**
  在 render 前调用（挂载和更新时都会），返回对象更新 state 或返回 null。用于根据 props 派生 state。
- **`getSnapshotBeforeUpdate(prevProps, prevState)`**
  在最近一次渲染输出（提交到 DOM 之前）调用，返回值作为 `componentDidUpdate` 的第三个参数，用于捕获 DOM 信息（如滚动位置）。

新版生命周期顺序：

- **挂载**
  `constructor` → `getDerivedStateFromProps` → `render` → `componentDidMount`
- **更新**
  `getDerivedStateFromProps` → `shouldComponentUpdate` → `render` → `getSnapshotBeforeUpdate` → `componentDidUpdate`
- **卸载**
  `componentWillUnmount`



### 函数组件生命周期

> 函数组件本身没有生命周期概念，但通过 **Hooks** 可以实现类似的效果。

#### 1. `useEffect`

> `useEffect` 是最常用的副作用 Hook，可以模拟挂载、更新和卸载。它让函数组件能够执行在渲染过程之外的操作，如**数据请求、DOM操作、设置订阅/定时器、日志记录**等，**可以看作是[componentDidMount](https://www.google.com/search?q=componentDidMount&oq=useEffect+的作用是什么呢+csdn&gs_lcrp=EgZjaHJvbWUyBggAEEUYOdIBCDYyNTVqMGo0qAIAsAIB&sourceid=chrome&ie=UTF-8&mstk=AUtExfDm-KYSnTBFCTf-ajAwy6h7VPkHv1OJZd7CXUXR9xx1Am-y85RfmlMHa_sEL4rqgZ53Bh-9p3X4gUg7YQUzz1Ph2Lt8E0HgR0Tyf_cJ5e2L7CNfhM1taxj5BpBvGys_NItCJLdCPlUt9ALmdp4-4DzSaLzkvDPCjfuVNN1dpI2WWZI&csui=3&ved=2ahUKEwiCx_2Nu5eSAxW5sVYBHbLRIyMQgK4QegQIARAB)、`componentDidUpdate`和[componentWillUnmount](https://www.google.com/search?q=componentWillUnmount&oq=useEffect+的作用是什么呢+csdn&gs_lcrp=EgZjaHJvbWUyBggAEEUYOdIBCDYyNTVqMGo0qAIAsAIB&sourceid=chrome&ie=UTF-8&mstk=AUtExfDm-KYSnTBFCTf-ajAwy6h7VPkHv1OJZd7CXUXR9xx1Am-y85RfmlMHa_sEL4rqgZ53Bh-9p3X4gUg7YQUzz1Ph2Lt8E0HgR0Tyf_cJ5e2L7CNfhM1taxj5BpBvGys_NItCJLdCPlUt9ALmdp4-4DzSaLzkvDPCjfuVNN1dpI2WWZI&csui=3&ved=2ahUKEwiCx_2Nu5eSAxW5sVYBHbLRIyMQgK4QegQIARAC)这三个类组件生命周期方法的组合**，通过控制第二个**依赖数组参数**来决定执行时机

- **空依赖 `[]`** → 仅挂载/卸载时执行（模拟 componentDidMount /  componentWillUnmount）
- **无依赖** → 每次渲染后执行（慎用！易导致性能问题）
- **依赖变化** → 精确控制更新时机  模拟 `componentDidUpdate`



- useEffect(callback，[]): （设置了空数组）但没设置依赖 , **模拟 `componentDidMount`** 

  ```tsx
  useEffect(() => {
    // 数据请求、订阅
    return () => {
      // 清理（模拟 componentWillUnmount）
    };
  }, []);
  ```

  - 第二个参数传空数组 `[]`，副作用只会在首次渲染后执行一次

  - **只有第一次渲染完成后，才会执行callback** ，等价于 componentDidMount ，每一次视图更新完毕后，callback 不再执行
  - callback 中，获取的都是最新状态值

- useEffect(callback，[num]): (每次渲染后执行) 设置了依赖, 只有依赖变化才执行，**模拟 `componentDidUpdate`**

  ````tsx
  useEffect(() => {
    // 每次 count 变化后执行 callback
  }, [count]);
  ````

  - 数组当中还可以放多个状态值，**第一次渲染完毕后**，执行callback，
  - 当依赖的状态值，或多个状态值中 的一个发生改变，也会触发callback 执行，依赖的状态如果没有改变，组件更新的时候callback 不会执行
  - 如果 callback 返回了一个小函数，那么，该小函数会在第一次渲染完毕后把组件释放的时候把上一次返回的小函数执行

-  useEffect(callback，[num]): 在 `useEffect` 的返回函数中执行清理工作     **模拟 `componentWillUnmount`**

  https://raw.githubusercontent.com/Hashmite/PicGo/main/img/20251213141904694.png

  ```tsx
  useEffect(() => {
    const subscription = props.source.subscribe();
    return () => {
      subscription.unsubscribe(); // 组件卸载时调用
    };
  }, [props.source]);
  ```

- ### `useLayoutEffect`

  与 `useEffect` 类似，但它在 DOM 变更后**同步**调用，适合读取布局或触发重绘（类似 `componentDidMount` 和 `componentDidUpdate` 的同步版本）



📊 Hooks 与类组件生命周期精准对照表

| 类组件生命周期            | 函数组件实现方式                         | 关键说明                   |
| :------------------------ | :--------------------------------------- | :------------------------- |
| `constructor`             | `useState(() => ...)`                    | 惰性初始化，避免重复计算   |
| `render`                  | **整个函数体**                           | 每次渲染重新执行（纯函数） |
| `componentDidMount`       | `useEffect(() => { ... }, [])`           | 依赖空数组，仅挂载后执行   |
| `componentDidUpdate`      | `useEffect(() => { ... }, [dep])`        | 依赖变化时执行（含首次）   |
| `componentWillUnmount`    | `useEffect(() => () => { cleanup }, [])` | 返回清理函数               |
| `shouldComponentUpdate`   | `React.memo(Comp, compare)`              | 浅比较 props，避免重渲染   |
| `getSnapshotBeforeUpdate` | `useLayoutEffect`                        | 读取 DOM 布局前状态        |
| `componentDidCatch`       | ❌ **必须用类组件**                       | 函数组件无法实现错误边界   |

#### 2. uselayoutEffect  和 useEffect 的细节

- useEffect 必须在函数的最外层上下文中调用，不能把其嵌入到条件判断、循环等操作语句中
- useEffect 如果设置了返回值，则返回值必须是一个函数（代表组件销毁时触发）；如果经过async 修饰，返回都是一个实例，而不是函数
  -  此时，我们可以使用 Xxx.then(箭头函数)，即可忽略掉返回一个实例的问题

- 对于useEffect 来讲： 第一次真实DOM 已经渲染，组件更新会重新渲染真实的DOM
- 对于useLayoutEffect来讲，第一次真实DOM 还未渲染，遇到callback中修改了状态，视图立即更新，创建出来的虚拟DOM，然后和上一次的虚拟DOM 合并在一起渲染为真实DOM. 也就是此类场景下，真实DOM 只渲染一次，不会出现图像的闪烁

## useRef  & uselmperativeHandle 的使用



> 类组件中，我们基于ref 可以做的事情 ：
>
> 1. 赋值给一个标签，获取DOM 元素
> 2. 赋值给一个类子组件： 获取子组件实例（可以基于实例调用子组件中的属性和方法等）
> 3. 赋值给一个函数子组件： 报错（需要配合React.forwardRef 首先ref 转发，获取子组件中的某一个 DOM 元素）
>
> ref 的使用方式：
>
> - ref=‘box’; this.refs.box 获取（不推荐使用）
>
> - ref={x=>this.box=x}; this.box 获取
>
> - this.box=React.createRef(); 创建一个ref 对象
>
>   < ref={this.box}> this.box.current  获取DOM 元素

如何在函数中 使用ref 呢？

1. 基于 “ref={函数}” 的方式，可以把创建的DOM 元素（或者子组件的实例）赋值给box 变量
2. 也可以基于React.createRef() 创建一个ref 对象
3. 推荐使用useRef函数直接创建ref 对象，该方式也只能在函数组件中使用	 

​        let box=useRef(null)



- useRef ,  每一次在组件更新时，再一次执行该方法时，获取的还是第一次创建的对象（推荐使用）
- React.createRef(); 每一次在组件更新时，再一次执行该方法时，都会创建一个全新的ref 对象。比较狼垡资源





总结： 在类组件中，创建ref 对象，我们基于React.createRef 处理：但是在函数组件中，为了保证性能，我们应该使用专属的 useRef 处理



在父组件中调用子组件情况：

- 当调用的是函数子组件时： 需要将该函数组件放在React.forwardRef(函数组件名(props,ref)) 进行转发使用，父组件想要获取子组件中的某个元素可以直接在该属性上加入 ref={ref}

  -   **如果想获取子组件中的内部的属性和方法，需要使用useImperativeHandle（ref,()=>{return(返回的是什么都会被ref 对象获取到) }）**

- 当调用的时函数的类子组件时： 直接在函数父组件中 加入ref={子组件的变量或者属性} 即可获取对应 的子租价的属性和值

  

## useMemo 构建计算缓存



useMemo（callback,[依赖变量]）

- 第一次渲染的时候，callback 会执行，后期只有 依赖的状态发生变化，callback 才会再执行，每一次把callback 执行的返回结果赋值给 xxx
- useMemo 具备缓存的效果，再依赖的状态值没有改变，callback 没有 触发执行的时候，xx 获取的是上一次计算出来的结果（计算缓存）

![useMemo](https://raw.githubusercontent.com/Hashmite/PicGo/main/img/20251215213620020.png)

- 作用：是一个优化的Hook 函数， 如果函数组件中，又消耗性能、时间的计算操作，则尽可能的使用useMemo 缓存起来设置对应的依赖


## usecallback缓存函数引用

> useCallback 用于得到一个固定引用值的函数，通常用它进行性能优化 其基本使用格式：
>
> ​       const cachedFn = useCallback(fn, dependencies)
>
> - 组件第一次渲染，useCallback 执行， 创建一个函数 callback  ，组件后续每一次更新，判断依赖的状态值是否改变，如果改变，则重新创建新的函数堆，赋值给xxz， 如果没有改变或者没有设置依赖，则xxx获取的一直是第一次创建的函数堆，不会创建新的函数堆
> - 简单来讲： useCallback 可以保证，函数组件的每一次更新，不再把里面的小函数重新创建，用的都是第一次创建的

🌟 为什么需要它？

在函数组件中，**每次渲染都会重新创建内部函数**。若将此类函数作为 prop 传给子组件（尤其是 `React.memo` 优化的组件），即使函数逻辑未变，**引用变化也会触发子组件重渲染**。
`useCallback` 通过保持函数引用稳定，解决此问题。

useCallback不能乱用，并不是所有的组件内部的函数，都要其处理会更好

1. 虽然可以减少了堆内存的开辟，但是useCallback本省也有自己的处理逻辑和缓存机制，这个也消耗时间
2. 如何使用呢：
   1.  父组件嵌套子组件，父组件要把一个内部的函数，基于属性传给子组件，此时传入这个方法，我们基于useCallback 处理一下会更好 

诉求：

-  当父组件更新的时候，因为传递的子组件的属性仅仅是一个函数（特点：基本应该是不变的），所以不想再让子组件也跟着更新了
  - 第一条：传递的 子组件的属性（函数），每一次需要是相同的堆内存地址（是一致的），基于useCallback 处理
  - 第二条：在子组件内部也要做一个处理，验证父组件传递的属性是否发生改变，如果没有改变，则让子组件不能更新，有变化才需要更新，继承React.PureComponent 即可（shouldCompnentUpdate中对新老属性做了浅比较）  （该方法属于类组件）  ![](https://raw.githubusercontent.com/Hashmite/PicGo/main/img/20251217220539719.png)
  -  第三条：如果是函数组件，那么则是基于React.memo 函数，对新老传递的属性做比较，如果不一致，则执行函数组件，如果一致，则不让子组件更新  ![](https://raw.githubusercontent.com/Hashmite/PicGo/main/img/20251217221444795.png)

## 合成事件原理 - 事件传播

> - 例如： 一个容器中，很多元素都要在点击的时候做一些事情
>  - 事件委托： 只需要给容器 做一个事件绑定（点击内部的任何元素，根据事件的冒泡传播机制，都会让同期的点击事件也触发，我们在这里，根据**事件源**，做不同的事情就可以了）
>   - 限制：当前操作的事件必须支持冒泡传播机制才可以
>     - 例如： mouseenter / mouseleave 等事件美誉偶冒泡 传播机制的
>   - 如果单独做到事件绑定中，做了事件传播机制的阻止，那么事件委托中的操作也不会生效！！！



事件具备传播机制，例如：当触发inner 的点击行为的时候

1. 从最外层向最里层逐一查找（捕获阶段：分析出路径）
2. 把事件源（点击的这个元素）的点击行为触发（目标阶段）
3. 按照捕获阶段分析出来的路径，从里到外，把每一个元素的行为也触发（冒泡阶段）   

从这里也能看出事件的传播机制是： **先捕获后目标再冒泡**



- ev.stopPropation(): 阻止事件传播，包含捕获和冒泡
- ev.stopImmediatePropagation() :立刻阻止当前的事件的其他方法未执行的不再执行（同级）

## 合成事件原理-事件委托



> 事件委托： 
>
>   利用事件的传播机制，实现的一套事件绑定的处理方案；根据**事件源target**，做不同的的事情就可以了



优势：

- 提高JS 运行的性能，并且把处理的逻辑都集中到一起
- 某些需求必须基于事件委托处理，
- 给动态绑定的 元素做事件绑定

### 合成事件原理

> 合成事件处理原理：
>
> - “**绝对不是**”给当前元素基于addEventListener 单独做的事件绑定，React 中的合成事件，都是基于“**事件委托**” 机型处理
>   - 在react17及以后版本，都是委托给 #root 这个容器（捕获和冒泡都做了委托）
>   - 在17版本以前，都是委托给document 容器的（且只做了冒泡阶段的委托） 
>   - 对于没有事件传播机制的这样的一个事件，才是单独做事件绑定 ；例如： onMouseEnter \ onMouseLeave       



**在组件渲染的时候**,如果发现JSX元素属性中有 onXxx/onXxxCapture 这样的属性,不会给当前元素直接做事件绑定,只是把绑定的方法赋值给 元素的相关属性!!例如:

```jsx
outer.onClick=() => {console.log('outer 冒泡「合成」');}

outer.onClickCapture=() => {console.log('outer 捕获「合成」');}
```



DOM 零级事件绑定是 outer.**onclick**, 而上述是大写的outer.**onClick** 绑定，故此，他不是给当前元素基于addEventListener 单独做的事件绑定，而它是基于事件委托处理的

然后对#root这个容器做了事件绑定「捕获和冒泡都做了」

​    **原因:**因为组件中所渲染的内容,最后都会插入到#root容器中,这样点击页面中任何一个元素,最后都会把#root的点击行为触发!! 而在给#root绑定的方法中,把之前给元素设置的onXxx/onXxxCapture属性,在相应的阶段执行!!

**其基本执行源码如下所示：** 

```jsx
< script >

const root = document.querySelector('#root'),
      outer = document.querySelector('#outer'),
      inner = document.querySelector('#inner');

// 经过视图渲染解析,outer/inner上都有onXxx/onXxxCapture这样的属性

outer.onClick = () => { console.log('outer 冒泡「合成」')}
outer.onClickCapture = () => { console.log('outer 捕获「合成」')}
inner.onClick = () => {console.log('inner 冒泡「合成」')}
inner.onClickCapture = () => {console.log('inner 捕获「合成」')}

// 给#root做事件绑定
    
root.addEventListener('click', (ev) => {
    let path = ev.path; // path: [事件源->....->window] 所有祖先元素

    [...path].reverse().forEach(ele => {

        let handle = ele.onClickCapture;
        if (handle) handlex();  // handlex(),如果不经过处理，方法中的this 是undefined,如果是箭头函数，则找函数上下文中的this
      });
}, true);

root.addEventListener('click', (ev) => {
    let path = ev.path;

    path.forEach(ele => {

        let handle = ele.onClick;
        if (handle) handle();
       });
}, false);
    
</script>
```



1. 视图渲染的时候，遇到合成事件绑定并没有给元素做事件绑定而是给元素做了设置对应的属性（合成事件属性）
2. 给#root做了事件绑定 ，在#root 上绑定的方法执行，把所有规则的路径中，有合成事件属性的都执行即可（捕获和冒泡）



所谓合成事件绑定，其实并没有给元素本身做事件绑定，而是给元素设置 onXxx/onXxxCapture 这样的合成事件属性。当事件行为触发，根据原生事件传播机制，都会传播到#root 容器上， react 内部给#root 容器做了事件绑定（捕获&冒泡）

当react内部绑定的方法执行的时候，会根据ev.path 中分析的路径，依次把对应阶段的onXxx/onXxxCapture 等事件合成属性触发执行 ====> 合成事件是利用事件委托（事件传播机制）完成的！！！

![](https://raw.githubusercontent.com/Hashmite/PicGo/main/img/20251206224532766.png)

### 合成事件机制

1. ev.stoPropagation():  合成事件中的“阻止事件传播”, 阻止原生的事件传播 & 阻止合成事件中的事件传播。 
2. ev.nativeEvent.stopPropagation(); 原生事件对象中的“阻止事件传播”；只能阻止原生事件 的传播



在16版本中，合成事件的处理机制，不再是把事件委托给#root元素，而是委托给docuent 元素，并且只做了冒泡阶段的委托，在委托的方法中，把 onXxx/onXxxCapture 合成事件属性进行执行

- 属性规则校验
  -  PropTypes from ‘prop-types’     类组件中父子组件之间的传值基于属性props 属性完成，配合属性规则校验使用

****

@1  ： 就是以父组件为主导,基于“属性”实现通信

原因:只有父组件可以调用子组件,此时才可以基于属性,把信息传递给子组件

- 父组件基于属性,可以把信息传递给子组件「父->子」
- 父组件基于属性「插槽」,可以把HTML结构传递给子组件「父->子」
- 父组件把方法基于属性传递给子组件,子组件把传递的方法执行「子->父」

@2： 父组件基于ref获取子组件实例「或者子组件基于useImperativeHandle暴露的数据和方法」

### 函数组件通信

> 函数组件也是通过props 进行通信，语法和类组件中的使用变了，函数组件中需要标注props 形参，在 React 函数组件中，父子通信完全基于 **props 传递 + 回调函数**，结合 Hooks 实现简洁高效的数据流。这里可以有两步优化

- 基于useMemo 实现复杂逻辑的“计算缓存”
- 基于useCallback 实现每次函数更新创建的地址为唯一地址值,依赖一定要加上
- memo 作用于函数组件上，做一个浅比较，导出去的时候之间把该函数抱起来



🔑 核心机制解析

| 通信方向     | 实现方式      | 关键代码                                        |
| :----------- | :------------ | :---------------------------------------------- |
| **⽗ → ⼦**  | Props 传递    | `<ChildComponent message={parentMsg} />`        |
| **⼦ → ⽗**  | 回调函数触发  | `onAction('数据')` → 父组件 `handleChildAction` |
| **性能优化** | `useCallback` | 避免父组件重渲染导致子组件 props 频繁变化       |
| **类型安全** | `PropTypes`   | 运行时验证（生产环境推荐）或 TypeScript通信方向 |

**封装http请求**

```jsx
/*
 http([config])
   + url 请求地址
   + method 请求方式  *GET/DELETE/HEAD/OPTIONS/POST/PUT/PATCH
   + credentials 携带资源凭证  *include/same-origin/omit
   + headers:null 自定义的请求头信息「格式必须是纯粹对象」
   + body:null 请求主体信息「只针对于POST系列请求，根据当前服务器要求，如果用户传递的是一个纯粹对象，我们需要把其变为urlencoded格式字符串(设定请求头中的Content-Type)...」
   + params:null 设定问号传参信息「格式必须是纯粹对象，我们在内部把其拼接到url的末尾」
   + responseType 预设服务器返回结果的读取方式  *json/text/arrayBuffer/blob
   + signal 中断请求的信号
 -----
 http.get/head/delete/options([url],[config])  预先指定了配置项中的url/method
 http.post/put/patch([url],[body],[config])  预先指定了配置项中的url/method/body
 */

import _ from '../assets/utils';
import qs from 'qs';
import { message } from 'antd';

/* 核心方法 */
const http = function http(config) {
  // initial config & validate 「扩展：回去后，可以尝试对每一个配置项都做校验?」
  if (!_.isPlainObject(config)) config = {};
  config = Object.assign({
    url: '',
    method: 'GET',
    credentials: 'include',
    headers: null,
    body: null,
    params: null,
    responseType: 'json',
    signal: null
  }, config);
  if (!config.url) throw new TypeError('url must be required');
  if (!_.isPlainObject(config.headers)) config.headers = {};
  if (config.params !== null && !_.isPlainObject(config.params)) config.params = null;

  let { url, method, credentials, headers, body, params, responseType, signal } = config;
  // 处理问号传参
  if (params) {
    url += `${url.includes('?') ? '&' : '?'}${qs.stringify(params)}`;
  }

  // 处理请求主体信息：按照我们后台要求，如果传递的是一个普通对象，我们要把其设置为urlencoded格式「设置请求头」？
  if (_.isPlainObject(body)) {
    body = qs.stringify(body);
    headers['Content-Type'] = 'application/x-www-form-urlencoded';
  }

  // 类似于axios中的请求拦截器：每一个请求，递给服务器相同的内容可以在这里处理「例如：token」
  let token = localStorage.getItem('tk');
  if (token) headers['authorization'] = token;

  // 发送请求
  method = method.toUpperCase();
  config = {
    method,
    credentials,
    headers,
    cache: 'no-cache',
    signal
  };
  if (/^(POST|PUT|PATCH)$/i.test(method) && body) config.body = body;
  return fetch(url, config)
    .then(response => {
      let { status, statusText } = response;
      if (/^(2|3)\d{2}$/.test(status)) {
        // 请求成功:根据预设的方式，获取需要的值
        let result;
        switch (responseType.toLowerCase()) {
          case 'text':
            result = response.text();
            break;
          case 'arraybuffer':
            result = response.arrayBuffer();
            break;
          case 'blob':
            result = response.blob();
            break;
          default:
            result = response.json();
        }
        return result;
      }
      // 请求失败：HTTP状态码失败
      return Promise.reject({
        code: -100,
        status,
        statusText
      });
    })
    .catch(reason => {
      // 失败的统一提示
      message.error('当前网络繁忙，请您稍后再试~');
      return Promise.reject(reason); //统一处理完提示后，在组件中获取到的依然还是失败
    });
};

/* 快捷方法 */
["GET", "HEAD", "DELETE", "OPTIONS"].forEach(item => {
  http[item.toLowerCase()] = function (url, config) {
    if (!_.isPlainObject(config)) config = {};
    config['url'] = url;
    config['method'] = item;
    return http(config);
  };
});
["POST", "PUT", "PATCH"].forEach(item => {
  http[item.toLowerCase()] = function (url, body, config) {
    if (!_.isPlainObject(config)) config = {};
    config['url'] = url;
    config['method'] = item;
    config['body'] = body;
    return http(config);
  };
});

export default http;
```

                /// Suspense.fallback：在异步加载的组件没有处理完成之前，先展示的Loading效果！！
                return <Suspense fallback={<>正在处理中...</>}>
                    <Component {...props} />
                </Suspense>;
            }} />;
        })}
    </Switch>;
};
export default RouterView;
```

## 在组件中获取路由对象

```tsx
index.tsx
import App from '../App'
import Home from '../view/Home'
import About from '../view/About'
import {BrowserRouter ,Routes,Route，Navigate} from "react-router-dom";

/* 所有的理由匹配规则，放在<Routes> 中；
                  每一条规则的匹配，都是基于<Route>;
                    + 路由匹配成功，不再基于component or render 控制渲染组件，而是基于 element
                    + 不再需要Switch ,默认匹配成功，就不再匹配下面的
                    + 不再需要exact, 默认都是精确匹配
                  原有的<Redirect> 操作，被<Navigate to='/'  /> 代替;遇到<Navigate />组件，路由就会跳转，跳转到to 指定的路由地址， to 的值可以是一个对象： pathname 需要跳转的地址，search 问号传参 
                    + 设置 replace 属性，则不会新增立即记录，而是替换现有记录
                  */
// 一般路由都是这样配置： 
const BaseRouter =()=>(
    <BrowserRouter>
      <Routes>
          <Route path="/" element={<App />}>
              <Route path='/' element={<Navigate to='/home' />}/> // 重定向
              <Route path="/home" element={<Home />}></Route>
              <Route path="/about" element={<About />}></Route>
          </Route>
      </Routes>
    </BrowserRouter>
)
export default BaseRouter

该导出的<BaseRouter/> 组件，应该放在整个文件的根路径上，也就是createRoot 里,而 <Outlet />  则是存放了路由目的组件的位置的占位符(也即是说，该占位符如果放在App.tsx 文件中，当点击对应路由跳转到该指定的组件页面时该页面就会在App.tsx 中 该占位符替换显示)
```
